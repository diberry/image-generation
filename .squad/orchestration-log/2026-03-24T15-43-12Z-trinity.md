## Learnings

**Code architecture:**
- OOMError sits as top-level class before parse_args() for easy importability
- batch_generate stays in generate.py (not separate batch.py) per test import expectations
- except + finally coexist in one try block; except re-raises, finally still cleans up — correct pattern for "transform exception but guarantee cleanup"

**Exception safety:** finally block executes unconditionally on:
- Success path (normal completion)
- OOM exception (caught and re-raised)
- KeyboardInterrupt (uncaught but finally still runs)
- Any other exception (re-raised after finally)

**Memory management discipline:**
- Per-item flushing in batch_generate prevents cross-item VRAM accumulation
- flush only between items, not after last (already flushed by generate's finally block)
- Device parameter cleanly converted to cpu flag for generate() call

**Production readiness:** Both features tested with 31 new tests (17 batch + 14 OOM). All edge cases covered: empty batch, partial failure, all-fail, state clean after OOM. Error messages actionable. No leaks. Ready for production use.
