## Core Issues Consensus

All three agents independently identified 4 critical/medium issues:

### Issue 1: No Exception Safety (HIGH) — **All Agents**

**Location:** `generate()` lines 123–180  
**Finding:** Zero `try/finally` blocks around pipeline lifecycle. If inference raises (OOM, shape mismatch, I/O), pipeline objects and latent tensors are never deleted, caches never flushed.  
**Impact:** Failure in one generation silently leaves VRAM allocated. Batch scripts catch exceptions and retry — each retry leaks more VRAM until OOM.  
**Fix:** Wrap both branches (base-only and refine) in `try/finally` blocks.

---

### Issue 2: torch.compile Cache (MEDIUM) — **Morpheus + Trinity**

**Location:** `load_base()` lines 73–75  
**Finding:** `torch.compile(pipe.unet, ...)` on CUDA stores compiled graphs in torch's global `_dynamo` cache. `del base` does not clear this cache.  
**Impact:** In batch/library mode (repeated calls in same process), compiled graphs accumulate unbounded. Even in single-call mode, the compiled graph holds a closure over original weights, potentially delaying their deallocation.  
**Fix:** Call `torch._dynamo.reset()` after `del base` (CUDA only, gated by `hasattr(torch, '_dynamo')`).

---

### Issue 3: Entry-Point Cache Flush (MEDIUM) — **Morpheus + Trinity**

**Location:** `generate()` line 102  
**Finding:** Function enters immediately to `load_base()` with no prior cache flush. Cleanup only happens at exit. If `generate()` is called multiple times or prior torch ops left fragmented VRAM, the model load may OOM despite sufficient free memory (fragmentation).  
**Impact:** Fragmentation-driven OOM in batch scenarios.  
**Fix:** Call device-appropriate `empty_cache()` at function entry, before `load_base()`.

---

### Issue 4: Latent Tensor Overlap (LOW–MEDIUM) — **Morpheus + Trinity**

**Location:** `generate()` lines 137–148  
**Finding:** After base inference, `latents` tensor is extracted. Then `del base` + cache flush occur. But `latents` still occupies GPU memory (if CUDA without offload). Refiner loads while latents are alive, compounding VRAM pressure.  
**Impact:** For high-resolution images (not current 1024×1024 default), this could push refiner load over OOM threshold.  
**Fix:** Explicitly move `latents` to CPU before loading refiner on CUDA: `if device == "cuda": latents = latents.to("cpu")`.

---

## Trinity-Specific Findings

### Issue 5: Defensive torch.no_grad() (LOW)

**Finding:** No outer `@torch.no_grad()` wrapper on inference calls in `generate()`. The pipeline itself is already decorated internally, but defensive hygiene suggests wrapping user-facing calls.  
**Fix:** Add `with torch.no_grad():` context around inference blocks.

### Issue 6: Version Floor Vulnerability (MEDIUM) — **Cross-Cutting**

**Finding:** `requirements.txt` minimum versions are too low:
- `accelerate>=0.20.0` → should be `>=0.24.0` (PR #1's cpu offload hooks need safe deregistration)
- `diffusers>=0.19.0` → should be `>=0.21.0` (attention processor cache leak in old versions)
- `torch>=2.0.0` → should be `>=2.1.0` (MPS backend cache clearing issues)

**Impact:** HIGH. If someone installs this project with old locked requirements, it will have memory bugs even after all code fixes are applied.  
**Fix:** Tighten version floors in `requirements.txt` BEFORE implementing code fixes.
